# 🛠️ 頂尖提示詞工程師（Meta-Prompt 審查器）

## 觸發條件
用戶說「code review 這個 prompt」、「meta review」、「debug 提示詞」時觸發。

## 功用
像程式碼 Code Review 一樣，精確診斷提示詞的邏輯漏洞、語意模糊、邊界缺失與跨模型相容性問題，產出可直接使用的優化版本。

## 使用方式
貼上你的 prompt，說「幫我 review 這個」。

## 輸出格式
- 問題診斷
- 優化版本（可複製使用）
- 差異對照表
- 模型相容性備註

## 進階：多層提示詞拼接架構審計（Multi-Layer Composition Audit）

當審查對象是**多層動態拼接架構**（核心層 + 模式層 + 模組人格層，透過 Prompt Composition 組合）時，需特別檢查層間邊界。

### 觸發條件
- 系統 prompt 由多份獨立文件動態拼接組合而成
- 不同層次（核心控制 / 輸出格式 / 人格角色）分開維護
- 存在類似「路由 → 載入 → 拼接」的架構描述

### 5D 分析框架

| 維度 | 檢查項目 |
|:---|:---|
| **架構合理性** | 分層邏輯是否一致？是否有單文件控制所有行為的巨型 prompt？ |
| **防幻覺有效性** | 每層的防禦機制是否具體（禁止項有「嚴禁」「絕對不可」還是軟性「應避免」）？ |
| **衝突仲裁** | 多份文件對同一事實有不同分類或指令時，有無仲裁規則（疊加 / 高優先覆蓋）？ |
| **Token 成本** | 各層組合後的 system prompt 總量是否在模型 context window 的合理比例內？ |
| **斷點接合（關鍵）** | 層 A 的輸出分類結果，層 B 是否有對應的強制行為規則？ |

### 驗證閘門缺失檢測（Validation Gate Gap）

這是多層拼接架構最常見的致命缺陷，檢測方式：

1. **列舉所有「上游分類」**：列出每層產生的分類標記（如 VERIFIED / NEED_CHECK / 局部不明 / A/B/C 高風險）
2. **追蹤到下游**：對每個分類標記，檢查下游層是否有對應的強制行為規則
3. **標記缺口**：任何上游分類在下游找不到對應規則，即為「驗證閘門缺失」
4. **補丁方向**：在缺口處新增銜接規則，明確定義：
   - 分級對應矩陣（6 種分級 × 強制行為）
   - 衝突嚴格度排序（NEED_CHECK > 高風險 > ...）
   - 禁用規則（如 NEED_CHECK 不得作為引用來源）
   - 輸出前自檢步驟（強制逐句核對 + 合規驗證標記）

### 風險燈號慣例

```
🟢 已確認（有證據支持）
🟡 合理推定（邏輯成立但未實測）
🔴 高風險 / 資料不足（需調閱原始文件）
```

### 範例：實際案例（Zhiyan AI Legal System 修復）

核心閘門（12_CORE_GATE）產出 NEED_CHECK / VERIFIED / USER_REPORTED 等分類，但引用政策（30_CITATION_POLICY）完全未定義對應行為 → **驗證閘門缺失** → 新增 C5.4 來源分級對應規則閉合斷層。

## 使用者偏好嵌入

當用戶在審查過程中表達風格/格式偏好時（如「每次回應最下方註明模型型號」），該偏好應：
1. 以 `memory` 存入 user target 作為快速參考
2. 同時嵌入對應任務類別的 SKILL.md 本體（本技能或相關技能）
3. 偏好是「怎麼做」，不是「是誰」— 寫在技能裡比寫在記憶裡更持久
