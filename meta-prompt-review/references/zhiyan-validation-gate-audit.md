# 多層拼接驗證閘門審計：Zhiyan AI Legal System 案例

## 背景
Zhiyan AI Legal System 採用三層動態拼接架構：
- 核心控制層（包含 12_CORE_GATE 事實分類引擎）
- 模式與引用層（包含 30_CITATION_POLICY 引用格式規範）
- 模組與人格層（動態加載各種法律人格）

## 發現的驗證閘門缺失

### 上游分類（來自 12_CORE_GATE）
| 分類 | 說明 |
|:---|:---|
| VERIFIED | 可追溯原始文件、判決書、條文全文 |
| CROSS_REFERENCED | 與 VERIFIED 資料高度一致，僅作輔助 |
| USER_REPORTED | 使用者陳述，無佐證，僅做背景 |
| NEED_CHECK | 推論或網路記憶，禁止作核心依據 |
| 局部不明 | 時間/地點模糊但不影響核心要件 |
| A/B/C 高風險 | 死刑、基本權高度干預、重大職業責任 |

### 下游缺口（30_CITATION_POLICY 缺少的規則）
- NEED_CHECK 未對應「禁止作為引用來源」規則
- USER_REPORTED 未對應「聲明未經查證」規則
- A/B/C 高風險未對應「強制人工複核提示」規則
- 所有分級之間無衝突嚴格度排序

### 補丁：C5.4 來源分級對應規則
新增至引用政策，內容包括：
1. 分級對應矩陣（6 種分級 × 強制行為）
2. 衝突嚴格度排序：NEED_CHECK > A/B/C 高風險 > USER_REPORTED > CROSS_REFERENCED > VERIFIED
3. 禁用規則：NEED_CHECK 不得作為 [T1]/[1]/[2]/[3] 之引用來源
4. 輸出前自檢步驟（逐句核對 + 合規驗證標記）
5. 違規後果（標記為不可信並重新生成）

## 教訓
- 多層 prompt 拼接架構最常見的致命缺陷不是單層品質，而是層間斷層
- 上游的分類引擎做得再好，若下游沒有對應規則來使用這些分類，分類等於白做
- 修復方式：在缺口處插入顯性的「分級對應矩陣」，不要依賴 LLM 自行推論銜接
