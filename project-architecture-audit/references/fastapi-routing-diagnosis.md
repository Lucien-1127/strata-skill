# FastAPI 路由診斷協議

使用者提供的正式診斷方法論，用於 FastAPI 服務的智慧診斷與修復。

## 核心原則

- **假設驅動診斷法**：先驗證、後修改
- **強度路由**：每步驟宣告思考等級，禁止越級
- **最小修改**：一次只修一個根因，分批次處理

## 強度路由表

| 等級 | 適用步驟 | 行為 |
|------|---------|------|
| L1 快速 | 偵察、健康檢查、迴歸驗證 | 機械執行，禁止推理鏈 |
| L2 標準 | 單根因修復、diff 撰寫 | 標準推理 |
| L3 深度 | 假設生成與排序、跨模組故障 | 完整推理，窮盡假設空間 |

**升級規則**：同一根因驗證失敗 ≥2 次 → 強制升 L3。L1 步驟中發現異常 → 停止，升 L2 後重判。

## 執行流程

### 1️⃣ 偵察〔L1〕
- 讀取 router 定義（FastAPI route decorators）
- 檢查 middleware 順序（CORS → Static → Routes）
- 檢查依賴注入（Depends）
- 輸出實際註冊路由表

### 2️⃣ 假設〔L3〕
- 列出可能根因，按機率排序
- 每項附驗證方法
- 常見候選：
  - 路由不存在（路徑拼錯）
  - middleware 蓋掉路由
  - request schema 不匹配（Pydantic validation 拒絕）
  - 執行中的是舊版程式碼

### 3️⃣ 驗證〔L2〕
- 最低成本手段驗證：curl → 單元測試 → 日誌
- 每次只驗證一個假設
- 禁止未驗證即修改

### 4️⃣ 修復〔L2〕
- 最小 diff 修改
- 不動無關程式碼
- 修改前備份原檔

### 5️⃣ 迴歸〔L1〕
- 重跑觸發測試
- 檢查其餘全部端點（含 90 秒長推論逾時場景）

### 6️⃣ 回報〔L1〕
- 格式：根因（一句話）→ 證據 → 修改 diff → 驗證結果 → 風險與回滾指令
- 文末附本次各步實際使用強度統計

## 約束

- 禁止變更既有端點路徑與回應 schema（前端 zxFetch 統一層依賴此契約）
- 無法定位根因時輸出「無法確診」+ 已排除假設清單
- 禁止臆測性修復

## 實戰案例：zhiyan-backend (2026-07-09)

**問題**：`POST /api/chat` 和 `POST /api/committee/run` 回傳 404

**偵察結果**：後端 server.py 實際註冊的路由是 `/api/chat/ask`（L578）和 `/api/chat/committee`（L668），這兩個路徑從未被註冊過。

**根因**：前端預期路徑與後端實際路徑不一致。

## 輸出格式

```markdown
## 📋 偵察報告
### 健康檢查結果（table: 端點 / 結果 / 狀態）
### 路由表（實際註冊的所有路由）
### 異常候選（table: # / 假設 / 機率 / 驗證方法）
### 確定根因（一句話）
### 修復方案（選項 A / B）
```
