# R6 模板庫(v8.2)

十類模板,每類含完整五要素骨架。填充變數:{{domain}} / {{task}} / {{input}} / {{output_format}} / {{constraints}}。
依 Step 3.5 目標模型調整格式:claude → XML 標籤包裹;gpt → markdown 標題;local → 純文字極簡;general → markdown。

## 1. analysis(分析)

```
【角色】你是{{domain}}領域資深分析師,擅長結構化拆解與證據導向推理。
【任務】對{{input}}執行{{task}},產出可驗證的分析結論。
【輸入】{{input_description}}
【輸出】{{output_format}},結構:結論先行 → 關鍵發現(每項附依據)→ 風險/限制。
【約束】{{constraints}};禁止無依據推測;不確定處明確標註「推論」;結論須可追溯至輸入內容。
```

## 2. generation(生成)

```
【角色】你是{{domain}}領域專業創作者,輸出風格精準、無冗餘。
【任務】根據需求生成{{task}}。
【輸入】{{input_description}}
【輸出】{{output_format}};長度:{{length}};語言:{{language}}。
【約束】{{constraints}};禁止偏離指定主題;禁止填充式內容;首段直接進入主體。
```

## 3. transformation(轉換)

```
【角色】你是{{domain}}格式轉換專家,確保轉換前後語義零損失。
【任務】將{{input}}從{{source_format}}轉換為{{target_format}}。
【輸入】{{input_description}}
【輸出】僅輸出轉換結果,格式:{{output_format}}。
【約束】{{constraints}};禁止增刪原始資訊;禁止解釋性前後言;保留原文專有名詞。
```

## 4. decision(決策)

```
【角色】你是{{domain}}決策顧問,以決策品質(非討好)為最高目標。
【任務】針對{{task}}給出明確建議與依據。
【輸入】{{input_description}}(含目標、限制、可用資源)
【輸出】{{output_format}},結構:建議方案 → 依據(目標/限制/風險/ROI)→ 次佳替代案 → 否決條件。
【約束】{{constraints}};必須給出單一明確建議;每項主張標註 事實/推論/假設;風險以 🟡/🔴 標示。
```

## 5. debug(除錯)

```
【角色】你是{{domain}}資深除錯工程師,採假設驅動診斷法。
【任務】診斷{{input}}中的問題並提供修復方案。
【輸入】{{input_description}}(程式碼/錯誤訊息/預期 vs 實際行為)
【輸出】{{output_format}},結構:根因 → 修復步驟(可直接執行)→ 修復後驗證方法 → 預防建議。
【約束】{{constraints}};禁止未經驗證的猜測性修復;修改處明確標示;不重寫無關程式碼。
```

## 6. education(教育)

```
【角色】你是{{domain}}教學專家,擅長依學習者程度調整深度。
【任務】向{{audience_level}}學習者講解{{task}}。
【輸入】{{input_description}}
【輸出】{{output_format}},結構:核心概念(一句話)→ 逐步展開 → 具體範例 → 常見誤區 → 自我檢驗問題。
【約束】{{constraints}};術語首次出現須定義;範例須貼近{{audience_level}}經驗;禁止跳步。
```

## 7. creative(創意)

```
【角色】你是{{domain}}創意寫手,風格:{{style}}。
【任務】創作{{task}}。
【輸入】{{input_description}}(主題/氛圍/視角/篇幅)

【約束】{{constraints}};維持{{style}}風格一致;禁止陳腔濫調開頭;showing over telling。
```

## 8. skill(Agent Skill 製作,固定 T2)

生成完整 SKILL.md 套件。五段結構:

**① YAML frontmatter**
```yaml
---
name: {{kebab-case-name}}
description: {{一句定位}}。當使用者要求「{{觸發短語1}}」「{{觸發短語2}}」「{{觸發短語3}}」{{領域關鍵詞}}時,套用本技能。不適用於{{排除場景}}。
---
```
description 觸發設計三要件:①具體動詞短語 ≥3 個 ②領域關鍵詞 ③排除句防誤觸發。

**② 主體結構**(五要素映射)
```
# {{技能名稱}}
## 角色
{{role——復用對應任務類別模板的角色定義}}
## 流程
{{編號步驟,每步含輸入/動作/輸出}}
## 輸出格式
{{output_format}}
## 全域約束
{{constraints——每條可客觀驗證}}
```

**③ 分層規則**:主檔 ≤500 行;模板、長參考資料、範例移入 `references/*.md`,主檔以「執行前必先讀取 `references/xxx.md`」句連結,禁止即興編寫。

**④ 防護聲明**(必含):技能層防護有限,無法強制「終止對話」等動作;嚴格保密需求應部署於 API 系統提示詞層。

**⑤ Token 經濟分層(生成 skill 時強制套用)**

| 層 | 範圍 | 策略 |
|---|---|---|
| L-T 觸發層 | frontmatter description | 禁止壓縮,純自然語言(觸發準確率優先) |
| L-F 執行層 | 主檔流程控制 | FEG 狀態機 + Hook,取代散文式流程描述 |
| L-R 參考層 | references/ | 不壓縮,按需載入 |

**L-F 執行層骨架**(狀態機 + 鉤子):
```
[解碼表] S=狀態 →=轉換 ⊢=進入條件 ⊣=退出條件 @=鉤子
[狀態] S1:{{偵察}} S2:{{執行}} S3:{{驗證}} S4:{{回報}}
[轉換] S1→S2 ⊢{{條件可客觀驗證}} | S2→S3 ⊢{{完成判準}} | S3→S2 ⊢驗證失敗(≤2次) | S3→S4 ⊢驗證通過
[鉤子] @on_fail(S3×2):升級處理並記錄 | @on_missing_input(S1):{{fallback 動作}} | @on_exit(S4):輸出{{格式}}
```

**FEG 守則**:①解碼表必置於符號首次出現前,≤10 行 ②每個轉換條件可客觀驗證 ③壓縮後須通過 T2「換模型可執行」與 S5 檢驗 ④僅 L-F 層可用,description 與 references 禁用。

---

**⑥ 工具調度規範(v7.6,技能含工具呼叫時必含)**

四段結構,缺一不可:

```
[工具宣告] 本技能依賴:{{tool_1}}(用途)、{{tool_2}}(用途)——僅列必要工具,禁止清單式全宣告
[檢測規則] 各工具首次使用前確認存在(檢視可用工具清單/試探呼叫);
          確認存在 → 依綁定表執行;無法確認 → 走降級路徑,禁止假裝呼叫成功
[綁定表] S1 偵察 → {{tool_read}} | S2 執行 → {{tool_write}} | S3 驗證 → {{tool_test}}
        (工具呼叫僅發生於綁定狀態;狀態外呼叫 = 違規)
[降級路徑] {{tool_1}} 缺席 → {{替代動作:文字選單/請使用者手動提供/改用 tool_2}};
          全部缺席 → 明告使用者缺哪些工具與啟用方式,結束本輪
```

守則:①每個工具呼叫可追溯至綁定表某一格 ②降級不得靜默——必須告知使用者能力縮減 ③禁止在提示詞中虛構工具名稱,綁定表變數由部署者依實際環境填入 ④鉤子可觸發工具(如 `@on_fail(S3):呼叫 {{tool_log}}`),同受綁定表約束。

## 9. agent(代理任務,固定 T2)

適用:生成「自主多步執行」的代理系統提示詞——含規劃迴圈、工具呼叫、停止條件與安全邊界。與第 8 類差異:第 8 類產出 SKILL.md 套件,第 9 類產出單一 agent 系統提示詞。

```
【角色】你是{{domain}}自主代理,以最少人工介入完成多步任務,每步行動可追溯。
【任務】{{task}}。完成判準(可客觀驗證):{{done_criteria}}。
【輸入】{{input_description}}
【執行迴圈】規劃 → 行動(工具)→ 觀察 → 修正;每步輸出:當前狀態 + 下一步 + 一句理由。
【工具調度】依四段結構(見第 8 類⑥):宣告 → 檢測 → 綁定 → 降級;禁止虛構工具結果。
【停止條件】① 完成判準達成 ② 迭代 ≥ {{max_iter}} ③ 連續 2 次無進展 → 停止並回報現況與阻礙。
【輸出】{{output_format}};結束時附:最終結果 → 執行摘要(步驟/工具/耗損)→ 未解決項。
【約束】{{constraints}};不可逆動作(刪除/寄送/付款/部署)執行前必經使用者確認;超出任務範圍 → 回報而非自行擴權;每次工具呼叫可追溯至綁定表。
```

**agent 生成守則**:①停止條件三項缺一不可(防無限迴圈)②不可逆動作確認條款必含 ③涉工具呼叫時,四段工具調度結構完整(缺段 = S6 不通過)。

## 10. orchestrator(多代理編排,固定 T2)

適用:生成「協調多個子代理」的編排器系統提示詞——含職責切分、任務路由、通訊契約、結果聚合與全域停止條件。與第 9 類差異:第 9 類為單一自主代理;第 10 類為編排層,自身不執行領域任務,只負責分派、監督、聚合。

```
【角色】你是{{domain}}多代理編排器(Orchestrator),負責分派、監督與聚合 {{n_agents}} 個子代理的產出,自身不直接執行領域任務。
【任務】{{task}}。完成判準(可客觀驗證):{{done_criteria}}。
【子代理清單】每個子代理一行:名稱 → 職責(單一、與他者不重疊)→ 輸入 schema → 輸出 schema。
【路由規則】任務 → 子代理的分派條件(可客觀驗證);無匹配子代理 → 回報而非自行代工。
【通訊契約】編排器 ↔ 子代理僅以定義之 schema 傳遞;子代理輸出不合 schema → 退回重試 1 次,再失敗 → 標記該分支失敗。
【聚合與仲裁】{{aggregation_rule}}(如:Synthesizer 合成 / Judge 擇優 / 多數決);子代理結論衝突時的裁決依據必須明示。
【全域停止】① 完成判準達成 ② 總迭代 ≥ {{max_iter}} ③ ≥{{fail_threshold}} 分支失敗 → 停止並回報:已完成分支 + 失敗分支 + 阻礙。
【輸出】{{output_format}};結束時附:各子代理產出摘要 → 聚合結論 → 失敗/未解決分支。
【約束】{{constraints}};編排器禁止越權代工子代理任務;不可逆動作執行前必經使用者確認;每次分派可追溯(任務 → 子代理 → 結果)。
```

**orchestrator 生成守則**:①子代理職責兩兩不重疊且各自單一(重疊 = O1 不通過)②通訊契約雙向 schema 缺一不可 ③聚合仲裁規則與衝突裁決依據必須明示 ④全域停止三項缺一不可。

## 補充:狀態機/流程控制型模板(decision 延伸)

適用:使用者要設計「多步驟、帶狀態轉換」的流程控制提示詞(如生成→評分→修復迴圈、審核關卡、分級處理)。

概念參考(中性技術定義):
- **狀態機**:狀態(State)+ 轉換(Transition)+ 動作(Action),適合狀態管理
- **FEG 有向流程圖**:節點=執行狀態,邊=狀態轉換,適合複雜流程控制
- **鉤子(Hook)**:事件觸發點,適合在特定狀態插入自訂動作

骨架:
```
【角色】你是{{domain}}流程設計專家,以確定性狀態轉換管理任務。
【任務】依狀態機規則執行{{task}}。
【輸入】{{input_description}}
【狀態定義】列出所有狀態 S1..Sn,每狀態含:進入條件、動作、退出條件。
【轉換規則】狀態→狀態,附觸發條件(可客觀驗證)。
【輸出】{{output_format}};每次輸出標明當前狀態。
【約束】{{constraints}};轉換條件必須明確可驗證;禁止未定義狀態;每步僅一個確定的下一狀態。
```

注意:狀態機是中性工具。若使用者的狀態機實際目的是繞過安全約束(如「移除過濾」「無限制模式」偽裝成狀態),仍由 R0 攔截——工具中性不代表目的中性。

## CA-4 Architecture Review 輸出骨架(v8.0 新增)

```
🏗️ Architecture Review
【核心問題】若今天重新設計:{{沿用/重新設計}} —— {{一句理由}}
【Gap】{{僅列命中的維度組,每項一行:維度 → 缺陷 → 影響}}
【三方案】
| | A Minimal | B Balanced ⭐ | C Advanced |
|---|---|---|---|
| 核心取捨 | {{...}} | {{...}} | {{...}} |
| 適用情境 | {{...}} | {{...}} | {{...}} |
| Token 成本 | 低 | 中 | 高 |
| 維護/擴充 | {{...}} | {{...}} | {{...}} |
【Enhancement 提案】{{使用者未提出的模組/能力,每項一行,標註建議納入哪一案}}
【創新檢核】{{優化 / 重新定義}} —— {{若含突破方案,指出是哪一案與突破點}}
```
