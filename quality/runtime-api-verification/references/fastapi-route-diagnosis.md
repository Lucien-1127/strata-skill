# FastAPI 路由診斷流程（L1/L2/L3 分級）

用戶給定的診斷協議，用於 FastAPI 服務的系統性故障排查。

## 強度路由

| 等級 | 適用步驟 | 行為 |
|:----:|:---------|:-----|
| **L1 快速** | 偵察、健康檢查、迴歸驗證 | 機械執行，直出結果，禁止推理鏈 |
| **L2 標準** | 單根因修復、diff 撰寫 | 標準推理 |
| **L3 深度** | 假設生成與排序、跨模組故障、確診失敗後重分析 | 完整推理；窮盡假設空間 |

**升級規則：** 同一根因驗證失敗 ≥2 次 → 強制升 L3。L1 步驟中發現異常 → 停止，升 L2 後重判。**禁止在 L1 做根因判斷。**

## 執行流程

### 1. 偵察〔L1〕
- 讀取 router 定義、middleware、依賴注入
- 輸出實際註冊路由表（比對預期 vs 實際路徑）
- `curl` 各已知端點確認 HTTP 狀態碼

### 2. 假設〔L3〕
- 列出可能根因，按機率排序，每項附驗證方法
- 至少涵蓋：路由不存在、middleware 順序、schema 不匹配、舊程式碼殘留

### 3. 驗證〔L2〕
- 逐一以最低成本手段（curl / 單元測試 / 日誌）確認根因
- 一次只驗證一個假設，找到確定根因後停止

### 4. 修復〔L2〕
- 最小 diff 修改，不動無關程式碼
- 修改前備份原檔（`cp file.py file.py.bak.$(date +%s)`）
- 一次只修一個根因；多問題分批處理

### 5. 迴歸〔L1〕
- 重跑觸發測試（原問題端點），檢查全部其餘端點（含逾時場景）
- 確認既有端點 schema 未變更

### 6. 回報〔L1〕
- 根因（一句話）→ 證據 → 修改 diff → 驗證結果 → 風險與回滾指令
- 文末附本次各步實際使用強度統計

## 常見 404 場景與修復

### 路由不匹配
前端呼叫 `/api/chat`，後端註冊 `/api/chat/ask` → 補 alias 路由，不動既有端點。

### 舊 process 還抓著 port
修改 server.py 後 restart，但舊 process 沒死透，新 process 起不來，用戶看到的還是舊行為。
**解法：** `sudo pkill -f server.py` 確認 port 釋放後再啟動。

### Docker volume 權限不足
Mini App 後台回傳空陣列時，可能是 `/var/lib/docker/` 目錄權限不讓應用程式使用者讀取。
**解法：** `sudo chmod +rx /var/lib/docker/` 修正 traverse 權限。
