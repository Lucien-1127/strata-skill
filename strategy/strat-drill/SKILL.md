# 🔨 Drill 壓力檢核

對 Anchor+5D 的產出進行壓力測試：模擬反例、探索邊界狀況、標記資訊缺口，並給出風險燈號。此模組可獨立重複套用在不同任務上。

Drill 的核心不只在找出當前的漏洞，更在 **持續自我質疑** — 檢核本身也需要被檢核。這形成一個遞歸的動態演化閉環。

## When to Use

- 由 `strat-orchestrator` 在 Anchor+5D 完成後呼叫
- 需要驗證方案的穩健性
- 需要紅隊測試（找邏輯漏洞）
- 直接呼叫：「進行壓力檢核」

## Prerequisites

- `strat-anchor-5d` 已完成並產出五維評分

## How to Run

```python
skill_view(name="strat-drill")
```

**輔助參考**：`references/drill-patterns.md` — 反例產生思維模式（M1-M6）、邊界檢查表、燈號決策矩陣、遞迴擴展示例。其中 M6（結構性缺口診斷）特別適用於檢查規範/協議/狀態機的定義完備性。選用，但新手建議先讀。

## Quick Reference

| 檢測項 | 方法 | 輸出 |
|:------:|:-----|:-----|
| 自我質疑 | 挑戰當前結論的前提 | 質疑列表 + 是否觸發重新 Drill |
| 反例測試 | 提出 2-3 個反例 | 反例列表 + 影響評估 |
| 邊界探索 | 尋找極端條件 | 邊界條件 + 風險標記 |
| 資訊缺口 | 對比已知 vs 需知 | 缺口列表 + 優先級 |
| 遞迴擴展 | 從缺口/反例衍生新問題 | 次輪 Drill 觸發或收斂判定 |
| 燈號標記 | 風險分級 | 🟢/🟡/🔴 |

## Procedure

### Phase 0 — 自我質疑預掃描（Pre-Drill Meta-Questioning）

在進入具體測試之前，先問三個元問題，防止檢核本身的盲點：

```
Q1: 這個任務的前提假設是否經得起反駁？
    → 列出 1 個最脆弱的假設
Q2: Anchor 階段是否遺漏了重要的對立觀點？
    → 補充 1 個反向框架
Q3: 如果我對這個結論的信心是錯的，最可能的原因是？
    → 找出最可能推翻的證據缺口
```

**質疑觸發規則**：若 Q1/Q2/Q3 中任何一個的回答顛覆了任務前提，跳過 Drill 直接返回 Anchor 重新錨定。這防止檢核浪費在已失效的基礎上。

### Phase 1 — 反例測試

針對任務目標提出 2-3 個具體反例：

```
反例 1: {情境描述}
  → 影響: {對方案的衝擊}
  → 應對: {可能的緩解方式}

反例 2: {情境描述}
  → 影響: ...
  → 應對: ...

### Phase 1.5 — 弱點排查（Weakness Audit）

在反例測試後、邊界探索前，加入一道強制自我誠實檢查。目的是揪出分析中「硬湊」的段落：

```
遍歷分析文本，標記所有感覺「我知道這條有點硬湊」的段落。

對每個標記段落，二選一：
  ✅ 補強：補充具體論據、數據或邏輯鏈條，使其站得住腳
  ❌ 刪除：承認此段落無法被合理支撐，直接移除

輸出格式：
| 序號 | 段落摘要 | 判定 | 理由 |
|:----:|:---------|:----:|:-----|
| 1 | {段落大意} | 補強 | {補充了什麼} |
| 2 | {段落大意} | 刪除 | {為什麼無法補強} |
```

> **誠實原則**：標記「硬湊」不是失敗，而是對分析品質的負責。與其保留一條弱論證讓整體可信度下降，不如誠實地移除它。Drill 的立場是「寧缺勿濫」— 一條弱分析會污染整份報告。

### 2. 邊界條件探索

| 邊界類型 | 問題 | 結果 |
|:---------|:-----|:-----|
| 上限 | 若規模放大 10x 仍成立？ | 成立/需調整/崩潰 |
| 下限 | 若資源減半仍可行？ | 成立/需調整/不可行 |
| 時限 | 若時間壓縮至 1/3？ | 可行/需取捨/不可行 |
| 依賴 | 若關鍵假設不成立？ | 有備援/需重新評估/無備案 |

### 3. 資訊缺口標記

對比「已知資訊」與「完成任務所需資訊」：

```
缺口 1: {缺少的資訊}
  優先級: 高/中/低
  取得方式: {推論/查詢/實驗}
  信心: XX%

缺口 2: ...
```

### 4. 風險燈號

匯總為三色燈號：

| 燈號 | 條件 | 含義 |
|:----:|:-----|:------|
| 🟢 **綠燈** | 無重大反例、缺口可控 | 可進入 Forge 階段 |
| 🟡 **黃燈** | 有中等風險但可緩解 | 需補充分析後再 Forge |
| 🔴 **紅燈** | 有致命反例或關鍵缺失 | 退回 Anchor 重新錨定 |

### 5. 遞迴擴展（Post-Drill Recursive Extension）

檢核本身不應是一次性動作。從 Phase 1-3 發現的缺口和反例中提問：

```
從反例/缺口中是否衍生出新的待檢核問題？
```

若 YES，列出次輪問題並判斷是否觸發重新 Drill：

| 衍生問題 | 來源（反例/缺口/邊界） | 是否觸發重新 Drill |
|:---------|:-----------------------|:-------------------:|
| {新問題} | {來源} | 是/否 |

**收斂條件**：
- 所有衍生問題的嚴重性 ≤ 已標記缺口 → 遞迴收斂
- 任何衍生問題為 🔴 級 → 重新執行完整 Drill
- 遞迴次數 > 2 仍未收斂 → 標記 🔴 + 強制輸出部分結果

> 「最高境界」不是靜態終點，而是不斷被重新定義的動態狀態。Drill 的價值不在找到一個終極答案，而在每次質疑後催生更深的檢核。這正是閉環動態演化：問題 → 檢核 → 新問題 → 新閉環。

### 6. 輸出

```
## 🔨 Drill 壓力檢核報告

### 反例測試
- ...

### 邊界條件
- ...

### 資訊缺口
| 缺口 | 優先級 | 信心 |
|:----|:------|:----:|
| ...  | 高    | 60%  |

### 風險燈號
🟢/🟡/🔴 — {一句話總結}
```

## Pitfalls

- **Drill 不是 Forge**：找出問題就好，不要在此階段提出解決方案 — 那是 Forge 的事。
- **反例要是合理的**：極端反例（如「核戰爆發」）無助於決策。保持在任務場景的合理邊界內。
- **燈號要明確**：不要同時給 🟡+🟢。選擇一個明確的燈號，不確定时給 🟡。
- **可重複執行**：Drill 可以在不同階段多次執行（如 Anchor 後一次、Forge 後再一次）。
- **自我質疑過度**：Phase 0 的三個元問題目的是篩選前提漏洞，不是推翻一切。若每個任務都退回 Anchor，表示 Anchor 本身產出品質有問題。
- **遞迴爆炸**：Phase 5 遞迴擴展的衍生問題可能無窮。嚴格遵從收斂條件：遞迴 >2 次或嚴重性 ≤ 已知缺口時強制終止。
- **燈號漂移**：同一任務在不同遞迴輪次中，燈號不應無理由地劇烈變動（🟢→🔴→🟡）。若發生，檢查是否因為遞迴引入了不相關的變因。
