# Drill 壓力檢核模式庫

STRATA Drill 階段的輔助參考：常見的壓力測試模式、邊界條件檢查表、反例產生思維、與燈號決策矩陣。此文件是選用工具 — 直接呼叫 Drill 會自動套用這些模式，但掌握它們能讓你更快讀懂 Drill 報告或手動進行壓力測試。

## 反例產生思維

Drill 的核心是「從對立面看問題」。以下 5 種思維模式幫你生成有效反例：

### M1：前提否定
假設核心前提不成立 → 方案是否仍有價值？

| 前提 | 否定情境 | 影響評估 |
|:-----|:--------|:---------|
| 用戶有足夠技術背景 | 用戶是完全新手 | 方案需要降級至入門層級 |
| 數據在單一系統中 | 數據分佈於多個來源 | 需要整合層 |
| 預算充足 | 預算砍半 | 優先級重新排序 |

### M2：逆向思維
方案說「最好的做法是 X」 → 試證明「X 也是最壞的做法」

### M3：使用者背叛
假設使用者的行為與方案預期完全相反：不按步驟、不讀文件、刻意輸入邊界值。

### M4：環境漂移
方案在開發環境完美運行 → 放入以下環境：
- 網路不穩定（丟包率 30%）
- 系統資源不足（記憶體僅 25%）
- 同時有 10 個用戶操作衝突

### M5：時間迴旋
把方案放到時間軸上：
- 6 個月後，方案的假設是否仍然成立？
- 如果方案的核心技術在 1 年後被淘汰？
- 方案的副作用（技術債、vendor lock-in）在哪個時間點會開始顯現？

## 燈號決策矩陣

| 反例嚴重性 | 缺口可控性 | → 燈號 | 行動 |
|:----------|:----------|:-----:|:-----|
| 無致命反例 | 高（缺口可推論填補） | 🟢 | 進入 Forge |
| 無致命反例 | 低（缺口需查證） | 🟡 | 補充分析 → 再 Drill |
| 有 1 個中等反例 | 高 | 🟡 | 記錄緩解方案 → 進入 Forge |
| 有 ≥1 個致命反例 | 任意 | 🔴 | 退回 Anchor |
| 遞迴 >2 次未收斂 | 任意 | 🔴 | 強制輸出 + 標記未完成 |

### M6：結構性缺口診斷（定義完備性壓力測試）

當一個規範（技能、協議、狀態機）宣稱某行為存在，但只定義了**觸發條件**和**終止條件**，跳過了中間的**執行主體**時，使用此模式。三步診斷：

**Step 1 — 辨識空洞**：將「做 X」替換為三個具體問題：
- 做什麼？具體步驟是什麼？（空洞 1：優先序缺失 — 若多個對象同時符合，先處理誰？）
- 做多少？單次操作範圍是什麼？（空洞 2：範圍未定義 — 一次動一個還是全部？每次修後基準是否改變？）
- 做完了然後？若結果不如預期怎麼辦？（空洞 3：異常路徑未定義 — 回退、降級、報告什麼？）

**Step 2 — 命診空洞**：給每個空洞命名（如「維度選擇無優先序」、「無本輪最小修復範圍定義」、「最佳版本定義模糊」），讓診斷可追溯、可驗證。

**Step 3 — 提供完整替代規範**：不是補丁，而是完整的、可直接替換原規範的精確文字。包含：觸發條件、執行步驟、異常路徑、終止規則。

**適用場景**：
| 典型信號 | 空洞類型 |
|:---------|:--------|
| 「Max 2 rounds」但沒說每輪做什麼 | 執行主體缺失 |
| 「fix → rescore」但 6 個維度同時扣分 | 優先序缺失 |
| 「output best version」但沒定義 best | 比較基準缺失 |
| 「repair if < 4.5」但修後更低沒處理 | 回退規則缺失 |

**置信度標記**：若診斷完全來自規範結構分析（不需外部案例），可標記 ≥90% 信心 🟢。此模式本質是結構性的，不依賴特定輸入行為。

## 常見遞迴擴展範例

Drill Phase 5 遞迴擴展可能產生的衍生問題：

| 原始發現 | 衍生問題 | 擴展方向 |
|:---------|:--------|:---------|
| 效能瓶頸在 I/O | 是否需要非同步架構？ | 架構層 |
| 用戶輸入無法預測 | 是否需要輸入 sanitizer？ | 安全層 |
| 方案假設穩定的 API | API 變更的頻率和通知機制？ | 維運層 |
| 錯誤處理集中在頂層 | 中間層是否需要部分降級？ | 韌性層 |
| 記憶使用未設上限 | OOM 時的自動降級策略？ | 資源層 |
