# FEG（Finite Execution Graph）決策核心

> 來源：老闆知識庫 | 儲存日期：2026-07-06

## 結論

FEG（Finite Execution Graph）是一種高效的決策核心，旨在解決多模態代理在面對複雜指令時的混亂問題。透過清晰的結構與規則，FEG 能夠快速判斷並執行最合適的任務。

## 場景描述

在一個典型的使用場景中，代理面對使用者的多重請求時，可能會因為指令的模糊性而產生混亂。例如，當使用者同時要求閒聊與翻譯時，代理可能無法正確判斷優先順序，導致輸出不連貫。

## FEG 核心結構

FEG 的語法可分為三大部分：

1. **空間定義與約束** (`FEG{...}`)：定義決策的範圍與基本規則。
2. **觸發語境與行為規範** (`RULES{...}`)：設定行為的邊界與優先級。
3. **狀態映射與生命週期** (`STATES{...}`)：定義任務的終點與異常處理。

### 微觀拆解

| 元件 | 語法 | 功能 | STRATA 對應 |
|:-----|:-----|:-----|:---------|
| 空間宣告 | `FEG{D{A..G}\|V:VM?}` | 宣告決策域範圍 + 管線強制讀取狀態 | S0 狀態初始化 + δ1-δ3 校驗路由 |
| 規則層 | `RULES{防呆 優先級}` | 邊界設定、意圖不明時的降級策略 | 約束鉤子 + δ2 衝突降級 |
| 狀態層 | `STATES{正常 異常 閉環}` | 終止條件、降級路徑、循環收斂 | FSM S3 品質閘門 + E1-E7 錯誤分類 |

### 管線強制讀取機制

FEG 的 `|->{X}` 語法強制每個決策域在執行前讀取當前狀態快照。這與 STRATA FSM 的 `每次狀態轉移記錄 key_variables` 設計目的相同 — 防止 E7 交接失真。

## 對應 strat 系統

- `FEG{D{A..G}}` 決策域分割 → strat-orchestrator 任務分類 + FSM 五狀態
- `RULES{防呆}` → δ1-δ3 安全校驗（δ1 診斷掃描、δ2 衝突降級、δ3 風險評估）
- `RULES{優先級}` → strat-anchor-5d 鉤子優先級（約束 > 語義 > 流程 > 結構）
- `STATES{異常降級}` → strat-forge-output fallback 契約 + E1-E7 錯誤分類
- `STATES{閉環}` → strat-drill Phase 5 遞迴擴展 + 收斂條件

## 設計建議

在設計多模態代理時，應重視 FEG 的結構化設計，並確保每個域的定義清晰，以避免混淆與錯誤執行。

## 風險

若未妥善設計 FEG 的規則與狀態，可能導致代理在面對複雜請求時無法正確反應，影響使用者體驗。

## 替代方案

考慮使用其他決策樹模型或基於機器學習的意圖識別系統，作為 FEG 的補充或替代。

## 整合點（與 STRATA V13.0）

1. **FEG 域宣告可作為 S1 任務分類的輸入格式** — 將模糊的非結構化查詢映射到預定義決策域（D_ANALYSIS, D_GENERATION, D_REWRITE, D_PLAN）
2. **RULES 防呆層 → δ1-δ3 的前置過濾** — 在語意校驗之前先用語法規則過濾掉明顯不合規的輸入
3. **STATES 降級路徑 → S3 品質閘門的 fallback 協議** — 統一兩個架構的降級行為，避免「降級不一致」造成使用者困惑

## 視覺化建議

建議將 FEG 的各個域與狀態映射放入 Canvas 白板，以便於視覺化理解其運作流程。
