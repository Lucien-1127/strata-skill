# 子代理派發 — QC 閉環

## 問題陳述

2026-07-07 session 中暴露的模式缺陷：
1. 派發子代理後 idle 等待，未利用時間繼續工作
2. 子代理回報後盲目信任摘要，未做 QC 就報告
3. 派發時未在 context 中內建驗證合約

## 正確流程

```
派發（context 含驗證合約）→ 繼續做其他事
    ← 子代理回報（含自檢點）
→ 主控抽檢 1-2 個關鍵點（非全文重讀）
→ 通過 → 回報用戶
→ 未過 → 退回修正
```

## 派發 context 模板

每次 delegate_task 的 context 中嵌入：

```
回報格式：
1. 完成摘要（2-3 句）
2. 修改的檔案路徑 + 行數範圍
3. QC 自檢點（你檢查了什麼、結果如何）
4. 未解決的問題（如果有）
```

## 驗證層級

| 層級 | 風險 | 驗證方式 | 範例 |
|:----:|:----:|:---------|:------|
| 低 | 例行、可逆 | 讀摘要即可 | 格式轉換、簡單查詢 |
| 中 | 有影響但可修正 | 抽檢 1-2 個關鍵 claim | 技能修改、文件產出 |
| 高 | 法律/合約/安全 | 完整驗證 + 獨立子代理審計 | 法律文件、合約審閱 |

## 相關鐵律

- H1 (¬HALLUCINATION)：子代理的 claim 必須有工具執行結果支撐
- H3 (QC)：回報前必須經過至少一層品質檢查
