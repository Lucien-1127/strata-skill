# 子代理 QC 模式（鐵律 H3 實作）

> 源自 2026-07-07 用戶糾正：「子代理說成功你就信了＝沒有 QC」

## 派工合約

dispatch 時，context 中應包含：

```
回報格式必須包含：
1. 完成摘要
2. 修改的檔案路徑 + 行數
3. 三個 QC 自檢點（你自己檢查的結果）
```

## 收工流程

1. 子代理回報 → **讀摘要**（不是全文重讀）
2. **抽檢 1 個 QC 點**（不是全驗）
3. 確認通過 → 回報用戶
4. 沒過 → 退回重做

## 不要乾等

dispatch 完成後，立即做別的事（回覆用戶、調研、調 config）。子代理自然回來，不需 polling。

## 關鍵反模式

- ❌ 子代理回報 → 直接信了 → 未驗證就回報用戶
- ❌ 派工後 idle 等待
- ❌ 全文重讀每一個子代理的完整輸出（只讀摘要 + 抽檢即可）
