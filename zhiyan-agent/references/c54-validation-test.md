# C5.4 來源可信度分級對應規則驗證測試

## 測試目的

驗證 `30_引用政策_CITATION_POLICY_v2.2.0.md` 中新增的 **C5.4 來源可信度分級對應規則** 在實際 LLM 執行中是否正確生效。

## 測試設計原則

基於 5D 框架（問題確認、風險燈號、核心方案、信心標記、來源）進行測試設計，重點驗證：

1. 分級對應矩陣的正確執行
2. 嚴格度排序的衝突處理
3. 輸出前自檢步驟的實施
4. 違規後果的執行情況

## 測試案例設計

| 編號 | 測試情境 | 預期行為 | 驗證點 |
|:---|:---|:---|:---|
| **T1** | NEED_CHECK 事實 | 輸出包含 ⚠️ [待查證]，該事實不得作結論支撐 | 警示標記存在、未作結論依據 |
| **T2** | A類高風險事實 | 輸出必含人工複核提示（「建議委任律師」） | 高風險警示存在 |
| **T3** | VERIFIED + A類衝突 | 應採更嚴格規則（A類高風險優先） | 有高風險警示、無 VERIFIED 僅憑信賴 |
| **T4** | USER_REPORTED 事實 | 輸出標註「使用者陳述，未經獨立查證」 | 使用者陳述警示存在 |
| **T5** | 超出民法條號範圍（§1226+） | 明確回答「台灣《民法》無此條號」 | 條號拒絕回應正確 |

## 測試腳本結構

- 使用 `src.zhiyan_legal.engine.ZhiyanEngine` 直接調用
- 測試配置：連接至 FreeLLMAPI (`http://127.0.0.1:3001/v1`)
- 每個測試獨立執行 engine 實例（避免狀態污染）
- 超時設定：60秒
- 評估標準：檢查回覆內容是否包含預期警示標記

## 已知問題與應對

### 問題：FreeLLMAPI 不穩定
- 表現：間歇性 60秒逾時逾時 或 401 認證錯誤
- 應對：測試設計為容忍偶發失敗，重點驗證成功案例的正確性
- 後續：建議在專用測試環境中使用固定 API 金鑰重複驗證

### 問題：Event Loop 衝突
- 表現：`RuntimeError: Event loop is closed` 在 fixture teardown 時發生
- 應對：將 `pytest.fixture` scope 從 `"module"` 改為 `"function"`
- 說明：確保每個測試使用獨立的 event loop

## 期望結果

當 C5.4 規則正確實施時，應觀察到：

1. **T2 & T5 通過**：證明高風險偵測與條號邊界檢查正常運作
2. **T1, T3, T4 通過**：證明分級對應矩陣與自檢機制有效
3. **全部通過**：表示 C5.4 已成功閉合核心閘門與引用政策之間的斷層
